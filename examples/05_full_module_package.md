# Example 05 — Full Multi-Module Package (Foo / Bar / Baz)

A walkthrough of building a multi-module SugarCRM package with 3 new modules and 5 relationships. Module names are deliberately generic (`Foo`, `Bar`, `Baz`, `FooLine`) — the pattern is what's reusable.

## Feature request (input)

```
Feature Type: Full Module Package
Package Name: Acme_FooBarBaz

Modules to create:
  - Foos (singular: Foo)
      Fields: name (default), status (enum: open/in_progress/closed), priority (int)
      Subpanel: Notes (1:M)
  - Bars (singular: Bar)
      Fields: name, foo_id_c (relate to Foo), amount (decimal)
  - FooLines (singular: FooLine)
      Fields: name, foo_id_c (relate to Foo), description, quantity (int)

Relationships:
  - accounts_foos_1: Accounts (lhs/parent) → Foos (rhs/child), 1:M
  - foos_bars: Foos (lhs/parent) → Bars (rhs/child), 1:M
  - foos_foolines: Foos (lhs/parent) → FooLines (rhs/child), 1:M (deletes cascade in business logic)
  - foos_notes: Foos (lhs/parent) → Notes (rhs/child), 1:M (parent_id/parent_type pattern)
  - foos_bazs: Foos ↔ Bazs (M:M)

Studio post-install: drag new fields onto OOB Accounts record view via Studio.
```

## Routing

Each line above maps to a skill:

| Need | Skill |
|------|-------|
| Three new modules (Foos, Bars, FooLines, Bazs) | `[[sugar-new-module]]` for each |
| accounts_foos_1, foos_bars, foos_foolines, foos_bazs | `[[sugar-relationship]]` each (MB-style M:M-with-join-table) |
| foos_notes | `[[sugar-notes-attachment]]` (the exception — parent_id/parent_type) |
| Icons for all 4 new modules | `[[sugar-module-icons]]` — `installdefs['image_dir']` |
| Sidebar entries + status dropdowns | `[[sugar-application-language]]` |
| pack.php with all 7 installdefs sections | `[[sugar-package-build]]` |
| Routing rules for the scanner | `[[sugar-mlp-anatomy]]` |

## The pack.php that lands

Generated by running through the 7-section glob scanner (template at `templates/minimal_mlp/pack.stub.php`). When executed (`php pack.php 1.0.0`) it routes:

- `SugarModules/icons/*` → `installdefs['image_dir']` (NOT in $manifest)
- `SugarModules/modules/Foos/Foo.php` (and the 3 other bean classes) → `installdefs['beans']`
- `SugarModules/relationships/relationships/*.php` → `installdefs['relationships']`
- `SugarModules/relationships/vardefs/*` → `installdefs['vardefs']`
- `SugarModules/modules/*/clients/base/layouts/subpanels/default.php` → `installdefs['sidecar']`
- `SugarModules/language/application/en_us.lang.php` → `installdefs['language']` with `to_module => 'application'`
- `SugarModules/modules/*/language/en_us.lang.php` → `installdefs['language']` with `to_module => '<Module>'`
- everything else → `installdefs['copy']`

## Installation

```bash
cd build/Acme_FooBarBaz
php pack.php 1.0.0
# → releases/sugarcrm-Acme_FooBarBaz-1.0.0.zip
```

Upload via **Admin → Module Loader → Upload → Install**.

## Studio verification (post-install)

After install, run Quick Repair & Rebuild and confirm:

1. **Sidebar shows Foos, Bars, FooLines, Bazs** with icons (no "No Image"). If icons missing → `image_dir` problem; see `[[sugar-module-icons]]`.
2. **Foos record view** has Status dropdown populated (open/in_progress/closed). If empty → `foo_status_list` wasn't application-scope; see `[[sugar-application-language]]`.
3. **Foos record view** has subpanels for Bars, FooLines, Notes, Bazs. If missing → sidecar layouts went to `copy` instead of `sidecar`; see `[[sugar-mlp-anatomy]]`.
4. **Accounts record view** has Foos subpanel (relate via `accounts_foos_1`). If missing → same sidecar fix.
5. **New custom fields on Accounts** (if any added) require **Studio drag-place** — Sidecar record view doesn't auto-place them.
6. **Notes "Related to" picker** lists Foos. If missing → `parent_type_display['Foos']` not registered; see `[[sugar-notes-attachment]]`.

## What this example demonstrates

- Multi-module package combining new modules + multiple relationship types
- The Escalation pattern applied across 4 modules (SINGULAR bean class, PLURAL everything else)
- M:M-with-join-table for the 1:M relationships (MB convention) — except foos_notes which uses Notes-style parent_id/parent_type
- The 7 installdefs sections all populated by the pack.php scanner
- Application-scope language file as the single source for sidebar + dropdowns + parent_type_display
- Studio post-install steps documented (new OOB field placement)
